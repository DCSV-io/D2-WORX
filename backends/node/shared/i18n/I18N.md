# @d2/i18n

Lightweight backend i18n package for loading JSON message catalogs and translating keys with parameter interpolation. Provides locale validation and resolution utilities shared across all Node.js services. Zero npm runtime dependencies (workspace dep: `@d2/service-defaults` for `parseEnvArray`; uses `node:fs` and `node:path`). Layer 0.

## Supported Locales

The canonical list of locales the platform supports, using **IETF BCP 47** tags. All locale files in [`contracts/messages/`](../../../../contracts/messages/) must match this list.

`BASE_LOCALE` is configurable via the `PUBLIC_DEFAULT_LOCALE` env var (defaults to `"en-US"`). The list is driven by the `PUBLIC_ENABLED_LOCALES__N` indexed env vars (parsed via `parseEnvArray`). If no env vars are set, it defaults to `[BASE_LOCALE]`.

```typescript
const BASE_LOCALE = (process.env["PUBLIC_DEFAULT_LOCALE"] ?? "en-US").trim();

// Default 10 locales (when all are enabled):
// en-US, en-GB, en-CA, es-ES, es-MX, de-DE, fr-FR, fr-CA, it-IT, ja-JP
```

### toBcp47

Normalises any locale tag to canonical BCP 47 casing: lowercase language, uppercase region. Bare language codes are lowercased.

```typescript
toBcp47("en-us"); // "en-US"
toBcp47("FR-CA"); // "fr-CA"
toBcp47("EN"); // "en" (bare language code)
```

### LANGUAGE_DEFAULTS

Maps bare language prefixes to their first matching BCP 47 locale. Built automatically from `SUPPORTED_LOCALES` (first locale per language prefix wins).

```typescript
// With the default 10 locales:
LANGUAGE_DEFAULTS = {
  en: "en-US",
  es: "es-ES",
  de: "de-DE",
  fr: "fr-FR",
  it: "it-IT",
  ja: "ja-JP",
};
```

### isValidLocale

Returns `true` if the locale string matches a supported BCP 47 tag. Input is normalized via `toBcp47()` before comparison.

```typescript
isValidLocale("en-US"); // true
isValidLocale("en"); // false (bare language code, not a full BCP 47 tag)
isValidLocale("EN-US"); // true (normalized via toBcp47 before comparison)
```

### resolveLocale

Safe resolver with **language-family fallback**. Resolution chain:

1. If the input is a valid supported locale, return it
2. If the input's language prefix matches a `LANGUAGE_DEFAULTS` entry, return that locale
3. Otherwise, fall back to `BASE_LOCALE` (`"en-US"`)

Handles `null`, `undefined`, empty string, and unsupported locale codes.

```typescript
resolveLocale("en-US"); // "en-US"
resolveLocale("en-GB"); // "en-GB" (exact match)
resolveLocale("en"); // "en-US" (language-family fallback via LANGUAGE_DEFAULTS)
resolveLocale("es"); // "es-ES" (language-family fallback)
resolveLocale("zh"); // "en-US" (unsupported — falls back to BASE_LOCALE)
resolveLocale(null); // "en-US"
resolveLocale(undefined); // "en-US"
```

## Translator

The `Translator` interface provides message translation with parameter interpolation. Created via `createTranslator()`.

### createTranslator

Factory that synchronously loads all `{locale}.json` files from a directory at startup. Locale keys are normalized to canonical BCP 47 casing via `toBcp47()`. Throws if the base locale file is missing or the directory does not exist.

```typescript
import { createTranslator } from "@d2/i18n";

const translator = createTranslator({
  messagesDir: "/absolute/path/to/contracts/messages",
  baseLocale: "en-US", // optional, defaults to BASE_LOCALE ("en-US")
});
```

**Options:**

| Option        | Type     | Required | Default   | Description                                              |
| ------------- | -------- | -------- | --------- | -------------------------------------------------------- |
| `messagesDir` | `string` | Yes      | --        | Absolute path to directory containing `{locale}.json`.   |
| `baseLocale`  | `string` | No       | `"en-US"` | Fallback locale. Must have a corresponding `.json` file. |

### Translator Interface

```typescript
interface Translator {
  /** Translate a key, with optional {param} interpolation. */
  t(locale: string, key: string, params?: Record<string, string>): string;
  /** All loaded locales (frozen array). */
  readonly locales: readonly string[];
  /** The base/fallback locale. */
  readonly baseLocale: string;
}
```

### Translation Lookup

The `t()` method normalises the input locale via `toBcp47()`, then resolves through a two-tier fallback chain:

1. Look up the key in the requested locale's catalog
2. If missing, look up in the base locale's catalog
3. If missing from all catalogs, return the key itself (enables safe pass-through)

Unknown locale codes fall through to base locale (step 2) without error.

### Parameter Interpolation

Messages can contain `{paramName}` placeholders. When a `params` object is provided, matching placeholders are replaced. Unmatched placeholders are left intact. Extra params not present in the message are silently ignored.

```typescript
// Message: "Hi {name},"
translator.t("en-US", "auth_email_verification_greeting", { name: "Alice" });
// → "Hi Alice,"

// Missing param — placeholder preserved
translator.t("en-US", "auth_email_verification_greeting", {});
// → "Hi {name},"

// Extra params — ignored
translator.t("en-US", "common_ui_save", { extra: "ignored" });
// → "Save"
```

### $schema Stripping

If a locale JSON file contains a `$schema` key (for JSON Schema validation in editors), it is automatically stripped during loading and not accessible as a message key.

## Message Catalogs

Locale files live in [`contracts/messages/`](../../../../contracts/messages/) at the monorepo root. Each file is a flat `{key: value}` JSON object named `{locale}.json` using BCP 47 tags:

- `en-US.json` (English -- United States, base locale)
- `en-GB.json` (English -- United Kingdom)
- `en-CA.json` (English -- Canada)
- `es-ES.json` (Spanish -- Spain)
- `es-MX.json` (Spanish -- Mexico)
- `de-DE.json` (German -- Germany)
- `fr-FR.json` (French -- France)
- `fr-CA.json` (French -- Canada)
- `it-IT.json` (Italian -- Italy)
- `ja-JP.json` (Japanese -- Japan)

Key naming conventions are defined in CLAUDE.md section 6 (Translation Key Conventions):

| Prefix                   | Scope             | Example                         |
| ------------------------ | ----------------- | ------------------------------- |
| `auth_{feature}_*`       | Auth pages        | `auth_sign_in_title`            |
| `webclient_app_{page}_*` | App pages         | `webclient_app_profile_title`   |
| `webclient_{section}_*`  | Design/demo/debug | `webclient_debug_session_title` |
| `common_ui_*`            | Shared UI labels  | `common_ui_save`                |
| `common_errors_*`        | Error messages    | `common_errors_not_found`       |

All locale files MUST stay in sync. When adding a key, add it to every locale file.

## Usage in Services

The translator is created once at service startup (composition root) and passed to components that need localized output (e.g., email/notification content in auth callbacks).

```typescript
// composition-root.ts
import { createTranslator } from "@d2/i18n";
import { resolveLocale } from "@d2/i18n";

const translator = createTranslator({ messagesDir });

// auth-callbacks.ts — translating notification content
const locale = resolveLocale(input.locale); // e.g., "en" → "en-US", "fr-CA" → "fr-CA"
const subject = translator.t(locale, "auth_email_verification_subject");
const greeting = translator.t(locale, "auth_email_verification_greeting", {
  name: userName,
});
```

The `resolveLocale` utility safely converts user-provided locale strings (which may be `null`, `undefined`, bare language codes, or unsupported) into a valid BCP 47 locale before passing to `translator.t()`. Bare language codes (e.g., `"en"`, `"fr"`) are resolved to their default regional variant via `LANGUAGE_DEFAULTS`.
