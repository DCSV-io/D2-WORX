# @d2/i18n

Lightweight backend i18n package for loading JSON message catalogs and translating keys with parameter interpolation. Provides locale validation and resolution utilities shared across all Node.js services. Zero runtime dependencies (uses only `node:fs` and `node:path`). Layer 0.

## Supported Locales

The canonical list of locales the platform supports. All locale files in [`contracts/messages/`](../../../../contracts/messages/) must match this list.

```typescript
const SUPPORTED_LOCALES = ["en", "es", "de", "fr", "ja"] as const;
type SupportedLocale = "en" | "es" | "de" | "fr" | "ja";
const BASE_LOCALE: SupportedLocale = "en";
```

### isValidLocale

Type guard that narrows `string` to `SupportedLocale`. Case-sensitive -- `"EN"` and `"jp"` return `false`.

```typescript
if (isValidLocale(locale)) {
  // locale is SupportedLocale
}
```

### resolveLocale

Safe resolver that returns the input locale if valid, otherwise falls back to `BASE_LOCALE`. Handles `null`, `undefined`, empty string, and unsupported locale codes.

```typescript
resolveLocale("es");        // "es"
resolveLocale("zh");        // "en" (unsupported — falls back)
resolveLocale(null);        // "en"
resolveLocale(undefined);   // "en"
```

## Translator

The `Translator` interface provides message translation with parameter interpolation. Created via `createTranslator()`.

### createTranslator

Factory that synchronously loads all `{locale}.json` files from a directory at startup. Files not matching a supported locale are skipped. Throws if the base locale file is missing or the directory does not exist.

```typescript
import { createTranslator } from "@d2/i18n";

const translator = createTranslator({
  messagesDir: "/absolute/path/to/contracts/messages",
  baseLocale: "en",  // optional, defaults to BASE_LOCALE ("en")
});
```

**Options:**

| Option        | Type              | Required | Default | Description                                             |
| ------------- | ----------------- | -------- | ------- | ------------------------------------------------------- |
| `messagesDir` | `string`          | Yes      | --      | Absolute path to directory containing `{locale}.json`.  |
| `baseLocale`  | `SupportedLocale` | No       | `"en"`  | Fallback locale. Must have a corresponding `.json` file.|

### Translator Interface

```typescript
interface Translator {
  /** Translate a key, with optional {param} interpolation. */
  t(locale: string, key: string, params?: Record<string, string>): string;
  /** All loaded locales (frozen array). */
  readonly locales: readonly string[];
  /** The base/fallback locale. */
  readonly baseLocale: string;
}
```

### Translation Lookup

The `t()` method resolves a message key through a two-tier fallback chain:

1. Look up the key in the requested locale's catalog
2. If missing, look up in the base locale's catalog
3. If missing from all catalogs, return the key itself (enables safe pass-through)

Unknown locale codes fall through to base locale (step 2) without error.

### Parameter Interpolation

Messages can contain `{paramName}` placeholders. When a `params` object is provided, matching placeholders are replaced. Unmatched placeholders are left intact. Extra params not present in the message are silently ignored.

```typescript
// Message: "Hi {name},"
translator.t("en", "auth_email_verification_greeting", { name: "Alice" });
// → "Hi Alice,"

// Missing param — placeholder preserved
translator.t("en", "auth_email_verification_greeting", {});
// → "Hi {name},"

// Extra params — ignored
translator.t("en", "common_ui_save", { extra: "ignored" });
// → "Save"
```

### $schema Stripping

If a locale JSON file contains a `$schema` key (for JSON Schema validation in editors), it is automatically stripped during loading and not accessible as a message key.

## Message Catalogs

Locale files live in [`contracts/messages/`](../../../../contracts/messages/) at the monorepo root. Each file is a flat `{key: value}` JSON object named `{locale}.json`:

- `en.json` (English -- base locale)
- `es.json` (Spanish)
- `de.json` (German)
- `fr.json` (French)
- `ja.json` (Japanese)

Key naming conventions are defined in CLAUDE.md section 6 (Translation Key Conventions):

| Prefix                    | Scope                        | Example                           |
| ------------------------- | ---------------------------- | --------------------------------- |
| `auth_{feature}_*`        | Auth pages                   | `auth_sign_in_title`              |
| `webclient_app_{page}_*`  | App pages                    | `webclient_app_profile_title`     |
| `webclient_{section}_*`   | Design/demo/debug            | `webclient_debug_session_title`   |
| `common_ui_*`             | Shared UI labels             | `common_ui_save`                  |
| `common_errors_*`         | Error messages               | `common_errors_not_found`         |

All locale files MUST stay in sync. When adding a key, add it to every locale file.

## Usage in Services

The translator is created once at service startup (composition root) and passed to components that need localized output (e.g., email/notification content in auth callbacks).

```typescript
// composition-root.ts
import { createTranslator } from "@d2/i18n";
import { resolveLocale } from "@d2/i18n";

const translator = createTranslator({ messagesDir });

// auth-callbacks.ts — translating notification content
const locale = resolveLocale(input.locale);
const subject = translator.t(locale, "auth_email_verification_subject");
const greeting = translator.t(locale, "auth_email_verification_greeting", {
  name: userName,
});
```

The `resolveLocale` utility safely converts user-provided locale strings (which may be `null`, `undefined`, or unsupported) into a valid `SupportedLocale` before passing to `translator.t()`.
